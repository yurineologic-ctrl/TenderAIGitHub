1. ОСНОВНЫЕ КОНЦЕПЦИИ
================================================================================

Git - система контроля версий
- Локальный репозиторий: папка .git на вашем компьютере
- Коммит: снимок вашего кода в конкретный момент (с описанием)
- Ветка (branch): отдельная линия разработки. Обычно "main" - основная ветка

GitHub - облачное хранилище репозиториев
- Удалённый репозиторий: копия проекта на серверах GitHub
- Push: загрузить коммиты на GitHub
- Pull Request (PR): предложение изменений для проверки

Workflow (рабочий процесс):
1. Вы меняете файлы локально
2. Коммитите изменения (git commit)
3. Пушите на GitHub (git push)
4. Создаёте PR (Pull Request)
5. Коллеги проверяют и мерджат в main

3. ТИПИЧНЫЙ ЦИКЛ РАЗРАБОТКИ
================================================================================

ШАГ 1: Создать новую ветку для фичи
  git checkout -b feature/название-фичи

Примеры имён веток:
  feature/add-gpu-detection
  fix/timeout-issue
  docs/update-readme

ШАГ 2: Делайте изменения в коде
  (просто редактируйте файлы в IDE)

ШАГ 3: Проверьте изменения
  git status          # Увидите какие файлы изменены
  git diff            # Увидите что именно изменилось

ШАГ 4: Добавьте файлы в staging (подготовка к коммиту)
  git add имя_файла.py

  Или добавить все файлы:
  git add .

  Или интерактивно (выбирать файлы):
  git add -i

ШАГ 5: Создайте коммит (сделайте снимок)
  git commit -m "feat: описание что вы сделали"

  Первая строка - краткое описание (до 50 символов)
  Вторая строка - пустая
  Остальное - детали (опционально)

ШАГ 6: Пушьте ветку на GitHub
  git push -u origin feature/название-фичи

ШАГ 7: Создайте Pull Request на GitHub
  - GitHub предложит кнопку "Compare & pull request"
  - Напишите описание изменений
  - Мережите PR в main (Merge pull request)

ШАГ 8: Вернитесь на main локально
  git checkout main
  git pull origin main

4. ПРАВИЛА ДЛЯ КОММИТОВ (Conventional Commits)
================================================================================

Начинайте каждый коммит с типа:

feat:     - новая фичя (feature)
fix:      - исправление ошибки (bug fix)
docs:     - изменение документации
style:    - форматирование кода (не меняет логику)
refactor: - переработка кода (не меняет функционал)
test:     - добавление/изменение тестов
chore:    - обновление зависимостей, конфиг и т.д.

ПРИМЕРЫ ХОРОШИХ КОММИТОВ:

git commit -m "feat: add support for GPU detection in parser"
git commit -m "fix: handle timeout errors in Playwright"
git commit -m "docs: update optimization guide with new tips"
git commit -m "refactor: extract regex patterns into separate module"
git commit -m "test: add pagination tests"

ПРИМЕРЫ ПЛОХИХ КОММИТОВ (НЕ ДЕЛАЙТЕ ТАК):
  git commit -m "changed stuff"
  git commit -m "bug fix"
  git commit -m "update"
  git commit -m "asdf"

5. РАБОТА С ВЕТКАМИ
================================================================================

Основная ветка (main) - это production-ready код.
Feature ветки - ваша текущая работа.

СОЗДАТЬ НОВУЮ ВЕТКУ:
  git checkout -b feature/add-gpu-detection

ПЕРЕЙТИ НА СУЩЕСТВУЮЩУЮ ВЕТКУ:
  git checkout main
  git checkout feature/add-gpu-detection

ПОСМОТРЕТЬ ВСЕ ВЕТКИ:
  git branch -a

  Вывод:
  * main                    (текущая ветка, отмечена *)
    feature/add-gpu-detection
    feature/fix-timeout

УДАЛИТЬ ЛОКАЛЬНУЮ ВЕТКУ (после мержа):
  git branch -d feature/add-gpu-detection

УДАЛИТЬ УДАЛЁННУЮ ВЕТКУ НА GITHUB:
  git push origin --delete feature/add-gpu-detection

6. ПРОВЕРКА ПЕРЕД ПУШЕМ НА GITHUB
================================================================================

1. ПРОВЕРЬТЕ СИНТАКСИС:
   python check_syntax.py

2. ЗАПУСТИТЕ ТЕСТЫ:
   python test_pagination.py

3. ПРОВЕРЬТЕ ИЗМЕНЕНИЯ:
   git diff
   git diff --staged

4. УБЕДИТЕСЬ ЧТО НЕ ЗАБЫЛИ НИЧЕГО:
   git status

5. ПОСМОТРИТЕ ЧТО БУДЕТ ПУШИТЬСЯ:
   git log origin/main..HEAD --oneline

================================================================================
7. ПОЛЕЗНЫЕ КОМАНДЫ
================================================================================

ПОСМОТРЕТЬ ИСТОРИЮ КОММИТОВ:
  git log
  git log --oneline           (компактный вид)
  git log --oneline -10       (последние 10 коммитов)
  git log --graph --all --oneline  (красивый график веток)

ПОСМОТРЕТЬ ИЗМЕНЕНИЯ КОНКРЕТНОГО ФАЙЛА:
  git diff имя_файла.py
  git log --oneline имя_файла.py

ОТМЕНИТЬ ПОСЛЕДНИЙ КОММИТ (но сохранить изменения):
  git reset --soft HEAD~1

ОТМЕНИТЬ ПОСЛЕДНИЙ КОММИТ ПОЛНОСТЬЮ:
  git reset --hard HEAD~1  ⚠️ ОСТОРОЖНО! Удаляет изменения

ПОСМОТРЕТЬ СТАТУС:
  git status
  git status -s    (короткий формат)

УВИДЕТЬ ЧТО БУДЕТ ПУШИТЬСЯ:
  git log origin/main..HEAD --oneline

ПОСМОТРЕТЬ ЧТО В КАКОЙ ВЕТКЕ:
  git branch -v

УДАЛИТЬ ФАЙЛ ИЗ КОММИТА БЕЗ УДАЛЕНИЯ ФАЙЛА:
  git reset HEAD имя_файла.py

8. РАСПРОСТРАНЁННЫЕ ОШИБКИ И КАК ИХ ИСПРАВИТЬ
================================================================================

ОШИБКА 1: Забыли добавить файл в коммит
  Решение:
    git add имя_файла.py
    git commit --amend       (добавляет файл в последний коммит)

ОШИБКА 2: Написали неправильное сообщение коммита
  Решение:
    git commit --amend -m "новое сообщение"

ОШИБКА 3: Запушили на неправильную ветку
  Решение:
    git push origin наименование_ветки:main  (пушить текущую ветку в main)
    Или создать PR и мержить оттуда

ОШИБКА 4: Сделали изменения в main вместо feature ветки
  Решение:
    git stash                (сохранить изменения временно)
    git checkout -b feature/название
    git stash pop            (вернуть изменения)

ОШИБКА 5: Конфликт при мерже (merge conflict)
  Решение:
    1. Откройте файл с конфликтом
    2. Отредактируйте конфликты (ищите <<<, >>>, ===)
    3. git add имя_файла.py
    4. git commit -m "resolve merge conflict"

================================================================================
9. ЗАЩИТА main ВЕТКИ (РЕКОМЕНДУЕТСЯ)
================================================================================

Чтобы никто не пушил напрямую в main:

1. Откройте GitHub репо
2. Settings → Branches
3. Add rule
4. Branch name pattern: main
5. Выберите:
   ✅ Require a pull request before merging
   ✅ Require status checks to pass before merging (если есть CI)

Теперь все изменения должны идти через PR!

10. АВТОМАТИЧЕСКИЕ ТЕСТЫ (GitHub Actions) - БУДУЩЕЕ
================================================================================

Можно добавить автоматические тесты при каждом пуше:

1. Создайте папку: .github/workflows/
2. Создайте файл: tests.yml
3. Напишите workflow который запускает тесты

Тогда при каждом пуше GitHub автоматически запустит тесты!

12. БЫСТРАЯ СПРАВКА (COPY-PASTE)
================================================================================

  git checkout -b feature/моя-фичя

  git push -u origin feature/моя-фичя

  git checkout main
  git pull origin main

  git status
